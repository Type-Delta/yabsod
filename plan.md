# YABSOD - Implementation plan

YABSOD (Yet Another Blue Screen of Death) is a CLI program that displays Windows' crash stats and application crash stats in a visually appealing way (within the terminal). It reads crash data from various Windows' native sources, such as the Event Log and the Reliability Monitor, using only PowerShell cmdlets and direct FS access. The program is written in Typescript and runs on Node.js for production environments, distributed via npm.

The core concept of YABSOD is a meme software that treats crashes as a feature and "embraces" it, all while providing actual useful information about the system's stability. The program will have a playful and humorous tone (gen-z like), with a focus on making the crash more enjoyable. With YABSOD, users can flex their crash stats and share them with friends instead of being ashamed. (As a meme, this program will treat crashes as something positive, and the opposite for stabale systems, which will be treated as "boring" and "uncool".)

## Features

### `yabsod stats`

`yabsod stats` is the main command of the program, which displays the crash stats in a visually appealing way (with unicode graphs).

here's what it will display:

- Total crashes (today/week/month/all-time)
- BSOD vs app crashes ratio
- Crashes per day average: github-like heatmap showing crash frequency for each day of the past months, the color of BSODs is blue and app crashes is red, if both happened on the same day, the color is mixed in proportion to the ratio of BSODs and app crashes
- Current uptime vs longest uptime streak (in %, with a progress bar)
- Most problematic hours: show top 3 hours of the day when crashes happen the most (with horizontal bar graph)
- "Days since last BSOD" counter
- Top 3 most crashed applications (with horizontal bar graph)
- Top 3 most frequent BugCheck codes (with horizontal bar graph)
- Top 3 processes that caused the most BSODs (with horizontal bar graph)
- "System stability score" that's perpetually 2/10
- Random sarcastic quotes based on crash count
- Achievement system: "100 BSODs! You've unlocked: Patience" (this is tied to `yabsod achievements` command) (achievements are unlocked but not viewed will be listed as "new", otherwise if the unlocked date is < 7 days ago, it will be listed as "recently unlocked")

By default, `yabsod stats` will show the stats for the past week, but users can specify a different time range (month, all-time) with a flag.
The display layout is grid-based, with sufficient terminal width, graphs may arrange horizontally, otherwise they will stack vertically.

### `yabsod jot`

Write crash events/dump into a sqlite database for later retrieval. This command is meant to be run in the background (e.g. as a scheduled task) to keep the crash data up-to-date for `yabsod stats`. It will read crash data from the Event Log and the Reliability Monitor, and store it in a local sqlite database.

This command has `--background` flag, which will reduce CPU resource usage at the cost of speed, for example disable parallel processing, this is meant to be used when running `yabsod jot` in the background (e.g. as a scheduled task), while without this flag, it will run faster but with higher CPU usage, which is meant to be used when the user wants to update the crash data immediately.

here's what the database will store:

- Crash timestamp
- Crash type (BSOD or app crash)
- Application name (for app crashes)
- BugCheck code (for BSODs)
- Process name (for BSODs)
- Report id (for BSODs)
- Report status (for BSODs)
- Stack trace (for BSODs)
- Path to related dump files such as Minidump, WER sysdata, WER reportqueue etc. (if available)
- Related Event Log entries (determined by timestamp proximity, source, level and other heuristics)
- OS version and build number, tied to each crash event (to track if a specific update caused more crashes)
- Drivers version tied to each crash event (to track if a specific driver update caused more crashes)
- Crash hash id (generated by hashing the crash event's timestamp, type, application name or BugCheck code and process name (if applicable) with a cryptographic hash function such as SHA256)

here's how it will work:

1. Cross-reference crash events from the Event Log and the Reliability Monitor with the database, if there are new crash events it will add them to the database.
2. For each new BSOD event, it will try to find the related dump files and Event Log entries, and store their paths and contents in the database.
3. It will also check if any new achievements have been unlocked based on the updated crash stats
4. Finally, prints a summary of what was done, new events added.

Note: if `yabsod jot` is run in TTY mode, it will display a spinner outlining the progress of each step, otherwise it will just print the summary at the end.

### `yabsod achievements`

This command displays the achievements that the user has unlocked based on their crash stats. Achievements are tied to specific milestones, each achievement will have a name, description and its own ASCII art icon. Completed achievements will be stored in the database with the timestamp of when they were unlocked, and can be viewed in this command. Some achievements can be "trimmed" to reach higher tiers, for example "Patience I" achievement can be upgraded to "Patience II", "Patience III" and so on, each tier will have its own description and icon but shared the same name with a tier suffix.
Achievements are updated whenever the user runs `yabsod jot`, which is responsible for updating the crash data in the database. When `yabsod jot` is run, it will check if any new achievements have been unlocked based on the updated crash stats, and if so, it will add them to the database, marked as achieved but not yet viewed. When the user runs `yabsod achievements` or `yabsod stats`, it will mark all unviewed achievements as viewed in the database, so they won't be shown as "new" anymore.
Some achievements have "after completion" stats that are only revealed after the achievement is unlocked. In case the record was broken after the achievement was unlocked, the "after completion" stats will show the new record instead of the record at the time of unlocking. (in the case that the achievement has multiple tiers, the "after completion" stats will only show after unlocking the highest tier)

With `-l` or `--list` flag, this command shows list of achievements with their name, description, after completion stats, icon and progress towards the next tier or completion (progress bar).
Otherwise, it will show only icons with name and progress or after completion stats in grid-based layout.
Achievements displayed will be separated into 3 sections: "Recently Updated" (recently unlocked or trimmed or broken old records), "Unlocked", and "Locked".
`--updated`, `--unlocked` and `--locked` flags can be used to filter the displayed achievements by their status.
`-f` or `--filter` flag can be used to filter the displayed achievements by name or description keywords. (fuzzy search)

### `yabsod list`

List crash events (application crashes and BSODs) with basic information such as timestamp, application name (for app crashes), BugCheck code (for BSODs) and process name (for BSODs). With various filtering options such as time range (mirror `git log`'s flags implementation), crash type, application name, BugCheck code, etc.
Each item is indexed with unique hash id, which can be used to view more details about the crash event with `yabsod view <id>` command, which will show all the information stored in the database for that crash event, including stack trace and related Event Log entries.
The hash id is pre-generated from `yabsod jot` and taking the first X characters of the hash as the id (X starts at 4 and increases when any duplicates are found). This ensures that each crash event has a unique and consistent id that can be used to reference it later.

### `yabsod view <id>`

View detailed information about a specific crash event, including stack trace and related Event Log entries. The crash event is identified by its unique hash id, which can be obtained from the `yabsod list` command. This command will display all the information stored in the database for that crash event in a readable format, with the stack trace and Event Log entries formatted for better readability.

The `<id>` can be the full hash or just the first few characters as long as it uniquely identifies a single crash event in the database, it can also be `~N` to reference the Nth most recent crash event. For example, `yabsod view ~1` will show the most recent crash event, `yabsod view ~2` will show the second most recent crash event, and so on.

If the crash event is a BSOD, it will also try to find and display the related dump files such as Minidump, WER sysdata, WER reportqueue etc. (if available) with their contents formatted for better readability. If the crash event is an app crash, it will try to find and display the related Event Log entries with their contents formatted for better readability.

For BSOD events, it will display the BugCheck description and possible causes based on the BugCheck code, using Microsoft's documentation as reference.

This command also accepts `--format` flag to specify the output format, avaliable formats are `default` (human-readable format with colors and formatting), `json` (machine-readable JSON format), and `md` (markdown format for sharing).

### `yabsod help`

A help command that shows the usage of the program and its commands, with examples and explanations for each command and its flags.

### Bug check code reference scraper

YABSOD needs bug check code descriptions and possible causes to display in `yabsod view <id>` for BSOD events and `yabsod jot` for achievements evaluation. To get this information the program will scrape Microsoft's documentation on bug check codes, which is available at [Bug check code reference](https://learn.microsoft.com/en-us/windows-hardware/drivers/debugger/bug-check-code-reference2). The scraper will extract the BugCheck code, its description and possible causes, and store them in a JSON file in project's repository for later retrieval. This scraper is run as a separate TS script on github actions that triggered once a month to keep the bug check code information up-to-date with Microsoft's documentation. The program will then download the latest bug check code information from the repository when needed, and cache it for a month before downloading it again.
